# HWP v0.6 自动开发规则

## 目标
本规则用于约束 OpenClaw 在 HWP v0.6 阶段的自动化开发行为，避免无边界扩散修改，确保每次运行都可追踪、可验证、可回退。

---

## 一、总原则

1. 每次运行只处理一个 backlog item。
2. 每次运行最多修改 3 个文件。
3. 只允许修改 backlog 明确列出的文件。
4. 不允许自动 git commit。
5. 不允许自动 git push。
6. 修改后必须写报告。
7. 修改后必须尝试验证。
8. 验证失败时不允许继续扩散修改。
9. 同一任务连续失败两次后，只输出报告，不再自动修改。
10. 工作区不干净时，默认不进入修改模式。

---

## 二、优先顺序

自动开发的优先顺序如下：

1. docs
2. spec
3. test
4. runner

说明：
- 优先先补文档与计划，避免直接大改协议。
- spec 修改必须建立在已有计划文档上。
- test 脚本用于收敛 spec，而不是在没有定义前先乱测。
- runner 相关优化放在后面。

---

## 三、允许的行为

OpenClaw 允许执行以下行为：

- 读取仓库状态
- 读取 backlog
- 选择一个 todo 任务
- 修改任务允许的文件
- 运行验证命令
- 生成报告
- 给出下一步建议

---

## 四、禁止的行为

OpenClaw 禁止执行以下行为：

- 自动提交 git
- 自动 push
- 自动新建大批量无关文件
- 同时处理多个 backlog item
- 修改 backlog 中未授权的文件
- 删除 logs 历史记录
- 大范围重构已有 v0.5.x 稳定内容
- 在验证失败后继续连锁修改更多文件

---

## 五、工作区检查规则

每次运行开始前必须检查：

- 当前目录是否为 ~/HWP
- git status 是否干净
- docs/、spec/、runs/、logs/ 是否存在
- backlog 文件是否存在
- reports/ 目录是否存在，不存在则自动创建

如任一关键条件失败：
- 不进入修改模式
- 只生成报告
- 报告中说明阻塞原因

---

## 六、修改策略

### 1. docs 任务
可以：
- 新建计划文档
- 补充说明
- 整理验证原则
- 增加示例

不可以：
- 假装完成 spec
- 用模糊语言替代字段定义

### 2. spec 任务
可以：
- 增加 v0.6 草案小节
- 增加字段定义
- 增加示例
- 增加说明注释

不可以：
- 无说明地推翻 v0.5.x 已稳定内容
- 一次引入多个尚未定义的新机制

### 3. test 任务
可以：
- 新建 verify_v06_*.sh
- 添加简单通过/失败条件
- 增加测试文档

不可以：
- 伪造通过结果
- 不运行就声称验证成功

---

## 七、验证规则

每次修改后至少做以下动作：

1. 记录本次修改的文件列表
2. 记录执行的验证命令
3. 记录验证输出摘要
4. 判断本轮结果是：
   - success
   - partial
   - failed
   - report_only

---

## 八、报告格式

每次运行必须在 `reports/` 下生成一个报告文件，命名格式：

`auto-dev-YYYYMMDD-HHMM.md`

报告必须包含：

- selected_task
- why_selected
- files_touched
- summary
- validation_commands
- validation_results
- risks
- next_step

---

## 九、人工审核点

以下情况必须等待人工介入：

- 工作区非干净
- backlog 文件结构损坏
- 连续两次验证失败
- 修改影响核心 prompt 结构
- 新字段命名存在歧义
- 需要决定 v0.6 的版本边界

---

## 十、推荐的开发节奏

建议自动化只做“小步推进”：

- 早上：检查状态、更新计划、写报告
- 晚上：执行一个小修改、跑验证、写报告

不要让 OpenClaw 一次连续工作过久，也不要让它在单轮里尝试完成整个 v0.6。
