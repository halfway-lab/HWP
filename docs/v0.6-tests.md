# HWP v0.6 测试设计文档

本文档定义 HWP v0.6 阶段的测试目标、验证方法和通过标准。

---

## 1. cross-round continuity 测试

### 1.1 测试目标

验证 HWP 多轮生成过程中的上下文连续性和状态一致性。

### 1.2 验证维度

| 维度 | 说明 | 检查点 |
|------|------|--------|
| 状态传递 | 前一轮输出正确传递到下一轮输入 | 状态字段完整性 |
| 语义一致性 | 跨轮次的语义组保持逻辑连贯 | 语义组标识符连续性 |
| 无信息丢失 | 关键信息在轮次间不丢失 | 关键字段存在性 |
| 增量正确性 | 每轮增量内容符合预期 | 增量字段验证 |

### 1.3 成功条件

- 状态文件在轮次间正确传递
- 语义组标识符跨轮次保持一致
- 关键字段（如 `round_id`, `continuity_score`）存在且有效
- 无异常中断或状态重置

### 1.4 失败条件

- 状态文件丢失或损坏
- 语义组标识符出现断裂或重复
- 关键字段缺失或格式错误
- 检测到意外的状态重置

### 1.5 执行方式

```bash
# 运行连续性验证
./runs/verify_v06_continuity.sh

# 或指定日志目录
./runs/verify_v06_continuity.sh /path/to/logs
```

### 1.6 Fixture 样本

测试 fixtures 位于 `tests/fixtures/` 目录：

| 文件 | 说明 | 样本数 |
|------|------|--------|
| `continuity_valid.json` | 合法样本 - 体现正确的跨轮连续性 | 4 |
| `continuity_invalid.json` | 非法样本 - 包含连续性违规情况 | 5 |

**正例样本包含:**
- Round 1: 初始状态 (30个变量, parent_id=null)
- Round 2: 健康的连续性 (25个共享变量, drift=0.17)
- Round 3: 强连续性适度漂移 (20个共享变量, drift=0.33)
- Round 4: 恢复后稳定 (parent_recovery_applied=true)

**反例样本包含:**
- Shared variables 断裂: 仅5个共享变量, drift=0.83
- Tensions 丢失: tensions 数组为空
- Chain 提前塌缩: Round 2 即 drift=0.67
- Parent 引用缺失: Round 3 的 parent_id=null
- 指标不一致: shared_variable_count 与实际不符

### 1.7 Fixture 接线说明

**Verifier 接收 Fixture 方式:**
```bash
./runs/verify_v06_continuity.sh --fixture [valid|invalid]
```

**Valid 通过条件:**
- `shared_variable_count` >= 10（Round 1 除外）
- `drift_rate` <= 0.6
- `tensions` 数组非空（Round 1 除外）
- `parent_id` 非 null（Round 1 除外）
- `shared_variable_count` 与实际变量数一致

**Invalid 失败条件:**
- `shared_variable_count` < 10 或 `drift_rate` > 0.6
- `tensions` 为空数组
- `parent_id` 为 null（非 Round 1）
- 指标不一致（`shared_variable_count` 超过变量总数）

## 2. semantic group expansion 测试

### 2.1 测试目标

验证语义组在扩展过程中保持内部一致性和逻辑连贯性。

### 2.2 验证维度

| 维度 | 说明 | 检查点 |
|------|------|--------|
| 组内一致性 | 同一语义组内的节点逻辑相关 | 语义相似度阈值 |
| 扩展边界 | 扩展不跨越不相关语义域 | 边界检测 |
| 标识稳定 | 语义组标识符在扩展中稳定 | 标识符连续性 |

### 2.3 成功条件

- 语义组内节点满足相似度阈值
- 扩展操作不引入跨域污染
- 组标识符在扩展前后保持一致

### 2.4 失败条件

- 组内节点相似度低于阈值
- 检测到跨域语义污染
- 组标识符异常变更

### 2.5 执行方式

```bash
# 运行语义组验证
./runs/verify_v06_semantic_groups.sh
```

### 2.6 Fixture 样本

测试 fixtures 位于 `tests/fixtures/` 目录：

| 文件 | 说明 | 样本数 |
|------|------|--------|
| `semantic_groups_valid.json` | 合法样本 - 体现正确的语义组扩展 | 4 |
| `semantic_groups_invalid.json` | 非法样本 - 包含语义组违规情况 | 5 |

**正例样本包含:**
- Single semantic group: 稳定扩展，coherence_score=0.92
- Multiple groups: 清晰边界，ethics/technical 分离
- Group expansion: 可追踪的血缘关系 (parent_node 引用)
- Stable identifier: 多轮扩展中 group_id 保持不变

**反例样本包含:**
- Group ID 为 null: 标识符断裂
- Group ID 意外变更: 标识符不稳定
- 扩展为噪音: coherence_score=0.45 低于阈值
- 跨域污染: cross_domain_contamination=true
- 不可追踪扩展: parent_node 引用缺失

### 2.7 Fixture 接线说明

**Verifier 接收 Fixture 方式:**
```bash
./runs/verify_v06_semantic_groups.sh --fixture [valid|invalid]
```

**Valid 通过条件:**
- `group_id` 为非 null 字符串
- `group_id` 保持稳定（无 `previous_group_id` 变更）
- `coherence_score` >= 0.5
- `expansion_path` 中的节点都在 `nodes` 中定义
- `parent_node` 引用有效（指向组内节点）
- `cross_domain_contamination` 为 false
- 无 `contaminated_domains`

**Invalid 失败条件:**
- `group_id` 为 null 或非字符串
- `group_id` 发生变更（`previous_group_id` 存在且不同）
- `coherence_score` < 0.5
- `expansion_path` 包含未定义的节点
- `parent_node` 引用无效
- `cross_domain_contamination` 为 true
- 节点存在 `contamination_flag=true`

## 3. blind spot detection 测试

### 3.1 测试目标

验证 blind spot detection 字段在 HWP 日志中的正确性和一致性。

### 3.2 验证维度

| 维度 | 说明 | 检查点 |
|------|------|--------|
| 字段存在性 | blind spot 字段是否存在于日志中 | 字段出现频率 |
| 类型正确性 | 字段类型是否符合 spec 定义 | 数组/数值/字符串 |
| 范围有效性 | blind_spot_score 是否在有效范围内 | 0.0 - 1.0 |
| 结构完整性 | blind_spot_signals 数组结构是否正确 | 对象字段完整性 |

### 3.3 成功条件

- 日志文件存在且可访问
- blind_spot_signals 字段存在且为数组类型
- blind_spot_score 字段存在且在 0.0-1.0 范围内
- blind_spot_reason 字段存在且为字符串类型
- 字段类型符合 spec 定义（非 null，非错误类型）

### 3.4 失败条件

- blind_spot_score 超出 0.0-1.0 范围
- blind_spot_signals 为 null 而非数组
- blind_spot_score 为字符串而非数值
- 字段类型与 spec 定义不符

### 3.5 执行方式

```bash
# 运行 blind spot 验证
./runs/verify_v06_blind_spot.sh

# 或指定日志目录
./runs/verify_v06_blind_spot.sh /path/to/logs
```

### 3.6 Fixture 样本

测试 fixtures 位于 `tests/fixtures/` 目录：

| 文件 | 说明 | 样本数 |
|------|------|--------|
| `blind_spot_valid.json` | 合法样本 - 符合 spec v0.6 定义 | 4 |
| `blind_spot_invalid.json` | 非法样本 - 包含各种违规情况 | 7 |

**正例样本包含:**
- Minimal: 无盲区情况 (score=0.0)
- Single information gap: 单一信息遗漏 (score=0.35)
- Multiple blind spots: 多个盲区高严重度 (score=0.75)
- Complete blind spot: 完全盲区 (score=1.0)

**反例样本包含:**
- Score 超出范围: 负值 (-0.5) 和 超上限 (1.5)
- 类型错误: signals 为 null、score 为字符串
- 字段缺失: 缺少 score 和 reason
- 枚举值错误: 无效 type 和 severity 值

### 3.7 Fixture 接线说明

**Verifier 接收 Fixture 方式:**
```bash
./runs/verify_v06_blind_spot.sh --fixture [valid|invalid]
```

**Valid 通过条件:**
- `blind_spot_signals` 存在且为数组（可为空）
- `blind_spot_score` 存在且为数值类型
- `blind_spot_score` 在 [0.0, 1.0] 范围内
- `blind_spot_reason` 存在
- Signal `type` 为有效值: `information_gap`, `logic_break`, `assumption_conflict`, `boundary_blindspot`
- Signal `severity` 为有效值: `low`, `medium`, `high`, `critical`

**Invalid 失败条件:**
- `blind_spot_signals` 为 null（非数组）
- `blind_spot_score` 为字符串类型
- `blind_spot_score` < 0.0 或 > 1.0
- `blind_spot_score` 或 `blind_spot_reason` 缺失
- Signal `type` 不在有效枚举中
- Signal `severity` 不在有效枚举中
